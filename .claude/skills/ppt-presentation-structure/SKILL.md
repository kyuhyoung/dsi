# 발표자료 슬라이드 표준 구성

## 슬라이드 종류 분류 (yaml `종류` 필드)

빌드 시스템(`yaml_to_pptx.py`)이 인식하는 종류:

| 종류 | 용도 | 한 발표당 권장 개수 |
|---|---|---|
| `표지` | 사업명·발주처·제안사·일자 표지 | 1 |
| `목차` | 발표 흐름 안내 | 1 |
| `간지` | **장(章) 시작 알리는 구분 슬라이드** | 장 개수만큼 (보통 5~8) |
| `회사소개` | 제안사 개요·핵심역량 | 1~2 |
| `본문` | 핵심 메시지·시각화 | 12~22 |
| `결론` | 요약·요청사항 | 1~2 |
| `부록` | Q&A·참고자료 (선택) | 0~3 |
| `감사` | 마지막 thanks 슬라이드 | 1 |

## 표준 슬라이드 순서 (간지 포함)

```
 1. 표지
 2. 목차
 3. 회사 소개 (1~2장)

 4. [간지] 1장. 사업 이해           ← 종류: 간지, 장번호: 1
 5. 사업 배경·목적                  ← 종류: 본문
 6. 추진 범위                       ← 종류: 본문

 7. [간지] 2장. 추진 방안           ← 종류: 간지, 장번호: 2
 8. 추진 전략
 9. 핵심 방법론
10. 단계별 접근

11. [간지] 3장. 추진 일정·조직      ← 종류: 간지, 장번호: 3
12. 일정 (간트차트)
13. 수행 조직 (조직도 + 핵심 인력)

14. [간지] 4장. 사업 수행 역량      ← 종류: 간지, 장번호: 4
15. 유사 실적
16. 보유 기술

17. [간지] 5장. 품질·보안 관리      ← 종류: 간지, 장번호: 5
18. 품질·보안 관리 본문

19. [간지] 6장. 결론·요청사항       ← 종류: 간지, 장번호: 6
20. 결론·요청사항
21. Q&A

22. [부록]                          ← 종류: 부록 (선택)
23. 감사합니다                       ← 종류: 감사 또는 결론(제목에 "감사" 포함)
```

## 간지(章 구분) 슬라이드 규칙 ⭐

**규칙 1.** 목차에 명시된 각 장(보통 5~8개) 앞에 간지 1장씩 필수 삽입.

**규칙 2.** 간지 yaml 필수 키:
```yaml
- 번호: 4
  종류: 간지
  장번호: '1'              # 또는 1
  장이름: '사업 이해'
  발표시간_분: 0.5
```

**규칙 3.** 간지를 빼면 청중이 장 전환을 못 느낌. 1분 미만 짧은 발표(<10슬라이드)에서만 생략 허용.

**규칙 4.** 간지의 `장이름`은 목차의 해당 장 제목과 *글자 그대로 동일*해야 일관성 유지.

**규칙 5.** 간지가 코드 fallback인 템플릿(견본 슬라이드 없음)에서는 자동으로 빈 슬라이드 + 큰 챕터번호·이름으로 렌더되므로 yaml만 넣으면 동작.

## 핵심 추출 원칙 (Essential Distillation) ⭐⭐

**제안서는 *읽는 문서*, 슬라이드는 *보는 도구*.** 본문 100을 슬라이드 100으로 옮기지 말고 본문 100을 슬라이드 20으로 *압축*한다.

### 슬라이드 1장 = 메시지 1개

각 슬라이드는 *하나의 결정·통찰·증거*만 전달. 메시지가 2개 이상이면 슬라이드 2개로 분리.

| 나쁨 | 좋음 |
|---|---|
| "사업 배경: 시장 규모 110조원이고 성장률 11.6%이며 디지털 전환 수요 확대" | slide A: "시장 110조원 (성장률 11.6%)" / slide B: "디지털 전환 수요 확대 → 진입 적기" |

### 5초 룰

청중이 슬라이드를 *5초만* 본다고 가정. 그 5초 안에 *핵심 한 문장*을 얻을 수 있어야 함. 못 얻으면 압축 부족.

- 슬라이드 제목 = 핵심 한 문장 (결론형)
- 본문 = 그 결론을 뒷받침하는 증거 1~3개 (불릿)

**제목 예 — 나쁨 (주제만):** "사업 배경"
**제목 예 — 좋음 (결론형):** "사업 배경 — 시험평가 자동화 적기"

### 결과 우선 (Lead with finding)

본문은 *배경 → 분석 → 결론* 순이지만, 슬라이드는 *결론 → 근거* 순으로 뒤집는다. 청중은 결론을 먼저 듣고 싶어함.

```
[본문]: "데이터 수집 → 모델 학습 → 검증 결과 F1-Score 0.98"
[슬라이드]: "F1-Score 0.98 (산업 표준 0.95 상회) ← 모델 학습 + 현장 검증"
```

### 정량 우선

수치가 있으면 *수치가 메시지*. 정성 표현 + 수치가 같이 있으면 수치를 제목에 끌어올린다.

| 나쁨 (정성 위주) | 좋음 (정량 위주) |
|---|---|
| "당사는 우수한 AI 기술을 보유하고 있습니다" | "AI 모델 F1-Score 0.98 (50개 농장 검증)" |
| "단기간 내 사업 완료 가능" | "12개월 일정 — 6개월 단축 (업계 평균 18개월)" |

### 항상 포함 (Must)

- **정량 차별성** — F1 0.98, 인력 64명, 매출 9억 등 본문의 *수치*
- **결론·요청사항** — "본 사업에서 우리가 무엇을 약속하는가" + "발주처가 무엇을 도와줘야 하는가"
- **시각요소 1개 이상** — 텍스트만 슬라이드 최소화

### 항상 컷 (Cut)

- **일반론·교과서적 설명** — "딥러닝이란~", "AI 기술 동향~"
- **반복 메시지** — 같은 차별성을 여러 슬라이드에 분산
- **배경 설명 길게** — 청중도 RFP 읽고 옴
- **타사 일반 정보** — 비교는 우리와 직접 관련 있을 때만
- **장식적 문구** — "당사는 최선을 다해...", "성공적으로 수행..."

### 1슬라이드 텍스트 임계

- 제목: ≤ 30자 (한 줄)
- 핵심메시지: 3개 × 30자 = 90자 이내
- 시각요소 캡션: ≤ 25자
- *합계 100자 초과 시 압축 부족 — 다시 자르거나 슬라이드 분리*

### 슬라이드 수 임계

- 발표 시간 ÷ 1.3분 = 슬라이드 수 (대략)
- 15분 발표 → 약 11~12장 (간지 제외 본문)
- 30분 발표 → 약 22~23장
- *이를 초과하면 메시지 중복 또는 압축 부족*

## 분량 가이드

| 항목 | 공공 사업 | 민간 사업 |
|---|---|---|
| 총 슬라이드 수 | 20~30장 | 15~25장 |
| 발표 시간 | 슬라이드당 1~1.5분 | 슬라이드당 1~1.5분 |
| 슬라이드당 핵심 메시지 | 3개 이내 | 3개 이내 |
| 슬라이드당 본문 글자 | 100자 이내 | 100자 이내 |

## 평가배점 ↔ 슬라이드 분량 매핑

발표 시간과 슬라이드 수는 평가배점 비율에 맞춰 배분:

- 평가배점 큰 항목 = 슬라이드 수 ↑, 발표 시간 ↑
- 예: "추진방안" 30점 → 슬라이드 7~10장, 발표 10~12분
- 예: "회사소개" 5점 → 슬라이드 1~2장, 발표 1~2분

## 슬라이드 레이아웃 vocabulary (필수)

각 본문 슬라이드는 `레이아웃` 키로 *명시적 배치*를 선언한다. 휴리스틱 의존 금지.

| 레이아웃 | 용도 | 필요한 yaml 키 |
|---|---|---|
| `text-only` | 텍스트·불릿만, 시각요소 없음 | `핵심메시지` 또는 `내용` |
| `image-focus` | 도면·다이어그램 1장이 메인 + 상단 1~2줄 설명 | `시각요소: [{type: image, path, caption}]` + 짧은 `핵심메시지` |
| `split-image-text` | 좌 이미지(55%) + 우 텍스트(45%). 도면 + 해설 | 이미지 1장 + 메시지 2~4개 |
| `split-text-image` | 좌 텍스트(45%) + 우 이미지(55%). 좌측 강조 | 메시지 + 이미지 1장 |
| `image-grid-2` | 이미지 2장 가로 비교 + 상단 짧은 메시지 | 이미지 2장 (각 caption) |
| `visual-only` | 차트·매트릭스·flow_arrow 등 시각요소 1개가 본문 거의 전체 | 시각요소 1개 + 메시지 0~2개 |
| (생략) | *기본 휴리스틱 모드* — 메시지+시각요소 자동 배치 | 호환성용. 신규 슬라이드는 명시 권장 |

### 레이아웃 선택 가이드

**이미지가 있는 슬라이드:**
- 이미지 1장 + 콘텐츠가 그 이미지의 *설명*이면 → `image-focus`
- 이미지 1장 + 콘텐츠가 *별도 정보*이면 → `split-image-text` 또는 `split-text-image`
- 이미지 2장 + 각각 caption 있어 비교·대조면 → `image-grid-2`
- 이미지 3장 이상이 한 슬라이드에 들어갈 것 같으면 → *슬라이드 분리* (1장씩 image-focus)

**이미지 없는 슬라이드:**
- 차트·매트릭스·timeline 등 시각요소가 메인이면 → `visual-only`
- 텍스트·불릿만이면 → `text-only`

### 슬라이드 분리 원칙 ⭐

복잡한 도면 여러 장을 한 슬라이드에 우겨넣는 대신 *분리*:

```yaml
# 나쁜 예 (이미지 2장 + 긴 텍스트가 한 슬라이드)
- 번호: 9
  제목: 핵심 AI 기능
  핵심메시지: [3개]
  시각요소:
    - {type: image, path: A.png}
    - {type: image, path: B.png}

# 좋은 예 (분리)
- 번호: 9
  제목: 핵심 AI 기능 (1/2) — AI 처리 흐름
  레이아웃: image-focus
  핵심메시지: ['멀티스케일 탐지, F1-Score 0.98']
  시각요소:
    - {type: image, path: A.png, caption: 5단계 AI 처리 파이프라인}

- 번호: 10
  제목: 핵심 AI 기능 (2/2) — 수행 조직
  레이아웃: image-focus
  핵심메시지: ['64명 R&D 조직 + 기업부설연구소']
  시각요소:
    - {type: image, path: B.png, caption: 다비오 R&D 조직표}
```

## 표(Table) → 차트 변환 가이드 ⭐

제안서 본문에 `|` 구분자 표 또는 `<표>` 표시가 있으면 *텍스트 셀로 옮기지 말고 차트로 변환*.

### 패턴별 변환 표

| 본문에 있는 표 패턴 | 변환할 시각요소 type |
|---|---|
| 항목 × 수치 (1~2 컬럼 비교) | `bar_chart` |
| 시계열 (년도/월/분기 × 수치) | `bar_chart` 또는 `timeline` |
| 단계별 일정 | `timeline` |
| 2×2 분류 (X축 × Y축) | `matrix_2x2` |
| 4~6 카테고리 + 짧은 설명 | `callout_cards` |
| 단순 비교용 행렬 (수치 없음) | `callout_cards` 또는 텍스트 표 |

### 변환 예시

본문 텍스트:
```
자기부담금 | 270.5백만원 | 30%
국고지원금 | 631.2백만원 | 70%
총 사업비   | 901.7백만원 | 100%
```

→ yaml:
```yaml
시각요소:
  - type: bar_chart
    title: 사업비 구성
    categories: ['자기부담금', '국고지원금']
    series:
      금액(백만원): [270.5, 631.2]
```

본문 텍스트:
```
2026 | 1.2억원
2027 | 3.5억원
2028 | 8.0억원
2029 | 15.2억원
2030 | 25.0억원
```

→ yaml:
```yaml
시각요소:
  - type: bar_chart
    title: 5개년 매출 계획
    categories: ['2026', '2027', '2028', '2029', '2030']
    series:
      매출(억원): [1.2, 3.5, 8.0, 15.2, 25.0]
```

### 변환 시 주의

- 수치 단위(억원·만원·%·명·건)를 series name 또는 title에 명시
- 행 4개 이상이면 가로 막대 vs 세로 막대 선택 (제목 길면 가로 권장)
- 표 셀이 *텍스트만*이면 (`낮음`/`중간`/`높음` 같은 정성 평가) callout_cards 사용
- 본문에 표를 *원본 그대로* 보여줘야 할 정당한 이유가 있을 때만 텍스트 표 유지

## 시각화 원칙

### 색상
- 3색 이내 (회사 CI 색 + 강조색 1개 + 중성색)
- 발주처 CI 색상은 *오해 방지*를 위해 사용 지양

### 폰트
- 2종 이내 (제목용·본문용)
- 발표장 뒷자리 가독성 기준 최소 18pt 이상

### 시각 요소
- 텍스트 위주 슬라이드 지양
- 다이어그램·플로우차트·표·이미지 적극 활용
- 모든 차트·표는 캡션 + 출처 명시

## 자주 하는 실수

- **빽빽한 텍스트** — 발표자가 읽는 슬라이드와 청중이 읽는 슬라이드가 충돌
- **작은 차트·표** — 뒷자리에서 안 보임
- **제안서 본문을 그대로 슬라이드에 옮김** — 재구성 필요 (본문 ≠ 슬라이드)
- **결론·요청사항 누락** — 마지막에 *"다음 단계로 무엇을 해주셔야 하는지"* 명시 필요
- **Q&A 슬라이드만 표지처럼 단독 배치** — 예상 Q&A 슬라이드를 부록에 준비해두면 가산점

## 출력 형식

ppt-designer agent가 이 skill을 참조해 슬라이드 구성안을 만들 때 사용할 형식:

```yaml
프레젠테이션:
  사업명: '...'
  발주처: '...'
  제안사: '다비오'

슬라이드:
  - 번호: 1
    제목: '표지'
    종류: 표지
    내용:
      사업명: '...'
      발주처: '...'
      제안사: '다비오'
      제출일: '...'

  - 번호: 2
    제목: '목차'
    종류: 목차
    내용:
      섹션:
        - {번호: '01', 제목: '사업 이해'}
        - {번호: '02', 제목: '추진 방안'}
        - {번호: '03', 제목: '추진 일정·조직'}
        # ...

  - 번호: 3
    제목: '회사 소개'
    종류: 회사소개
    핵심메시지: ['...']

  # 1장 간지 — 반드시 본문 슬라이드 앞에 위치
  - 번호: 4
    제목: '사업 이해'
    종류: 간지
    장번호: '1'
```

### yaml 작성 체크리스트

- [ ] `종류: 표지` 1개 (반드시 첫 번째)
- [ ] `종류: 목차` 1개 (섹션 N개 명시)
- [ ] `종류: 간지` N개 (목차의 섹션 수와 일치, 본문 그룹 앞마다)
- [ ] `종류: 본문` 슬라이드는 핵심메시지 1~3개 + 시각요소 1개 권장
- [ ] 마지막 슬라이드는 `종류: 결론` (제목에 '감사' 포함) 또는 `종류: 감사`
- [ ] 간지 `장이름` = 목차 섹션 제목과 글자 일치
