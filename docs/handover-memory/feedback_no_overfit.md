**원칙 (사용자 확정 2026-05-24)**: *"임의의 A, B, C에 대해 가능해야 한다. 현재 케이스(농식품AI)에 overfitting/하드코딩 금지."*

**Why**: 시스템 목적이 *임의 RFP→제안서 자동화*. 농식품AI 한 케이스만 잘 돌리는 솔루션은 가치 없음. 과거 커밋에 "농식품AI 하드코딩 0" 자체검증 문구가 박혀있을 만큼 *상시 위험*.

**How to apply**:
- **하드코딩 금지 대상**:
  - 양식 파일명·표 개수·셀 좌표·라벨 텍스트
  - RFP 사업명·발주처·예산·평가배점
  - KB 회사명·실적·정량지표
- **PY 코드 검사 체크**:
  - if/elif 분기 안에 "농식품"·"AI 응용제품"·"다비오"·"AgriVision" 같은 *고유명사* 박혀있나? → 즉시 yaml 또는 동적 추출로 옮김
  - 매직넘버 (표 개수 86·셀 좌표 등) 박혀있나? → 입력 yaml에서 읽음
- **스키마 일반성**:
  - form.yaml: 셀 id는 좌표 기반 (T{n}_R{r}_C{c}) — 도메인 무관
  - hints: 위/왼 셀 텍스트 *자동 추출* (의미 분류 하드코딩 X)
  - filled_cells.yaml: id 기반 — 사업/양식 무관
- **검증 방법**:
  - 코드 변경 후 *다른 .hwp 양식* (예: 민군규격, F16PBU) 으로 동일 흐름 시도해 동작 확인
  - 농식품AI ideal D diff는 *최종 단계*에서만. 그 전엔 일반성 검증 우선
